When Claude executes a tool (bash, read, etc.), is the system prompt re-injected alongside the tool result?	No. The tool result passes back to the model as bare output. System prompt and skill instructions are NOT re-injected with each tool result -- only on the next full API call.		internals::tool-execution-context
What is the highest-risk pattern for behavioral rule adherence during tool execution?	Multi-step bash chains (5-6 tool calls in a row). Each tool result arrives bare, and behavioral context from the system prompt is further from immediate reasoning at each step.		internals::tool-execution-context
During a workshop, someone asks: "I put a rule in my skill file to always confirm before destructive commands, but Claude ran cf delete without asking." What's the likely explanation?	Skill file instructions are part of the system prompt, sent on every API call, but NOT re-injected into individual tool result blocks. In a multi-step tool chain, those rules are less likely to be the focal point when processing tool results mid-chain. Fix: restate critical constraints in the user message for that specific task, or break long chains into shorter steps with explicit reasoning.		internals::tool-execution-context
What is the most reliable way to ensure Claude follows a rule during tool-heavy workflows?	Include the rule in the user prompt for that specific task, not only in the skill file or CLAUDE.md. This keeps the instruction close to the reasoning context during tool execution chains.		internals::tool-execution-context
How should you structure prompts for sensitive multi-step operations to avoid rules being forgotten mid-chain?	Break long tool chains into shorter steps with explicit reasoning between them. Instead of "Check all foundations, compare counts, flag anomalies, generate report" use separate prompts for each step so Claude reasons between tool calls with full context.		internals::tool-execution-context
