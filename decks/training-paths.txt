How many training paths does the wiki offer, and how are they organized?	Three paths organized by role: Developer Path (prompting through extensions), Platform Engineer Path (infrastructure through rollout), and Product Manager Path (product research through team collaboration). Role-based organization lets each audience focus on what matters for their work.		training::overview
How many modules does the Developer Path have, and what's the progression logic?	6 modules: (1) Prompting Foundations, (2) Verification Loop, (3) TDD, (4) Debugging, (5) Context Management, (6) Extending Claude Code. The progression moves from "how to talk to Claude" to "how to build feedback loops" to "how to manage the tool at scale." Each module depends on the previous one.		training::developer-path
How many modules does the Platform Engineer Path have, and what's the organizing principle?	6 modules: (1) Architecture, (2) Infrastructure, (3) Configuration and Policy, (4) Permissions and Security, (5) Cost Management, (6) Phased Rollout. Organized by deployment stack layers -- you understand the architecture first, build infrastructure, then layer policy, security, cost controls, and finally execute a rollout.		training::platform-path
How many modules does the PM training path have and what's the organizing principle?	5 modules. Product-first orientation: PMs learn to use Claude Code for their own work (research, requirements, prioritization) before learning technical literacy for developer collaboration.		training::pm-path
Explain why the Developer Path starts with prompting foundations instead of jumping into coding workflows.	Key points:<br>1. Prompt quality determines output quality -- vague requests produce vague results, so this is the foundational skill<br>2. Without prompt specificity (what, where, why), developers fight the tool instead of leveraging it<br>3. It's analogous to learning to communicate requirements before writing code -- the skill transfers to every subsequent module<br>4. Module 1 establishes the pattern of explicit instructions that Modules 2-6 build on (verification, TDD, debugging all depend on clear prompts)		training::developer-path
Explain why the Verification Loop is Module 2 in the developer path rather than being taught alongside prompting.	Key points:<br>1. Prompting tells Claude what to do; verification confirms it did the right thing. They're distinct skills that build on each other<br>2. The four-step loop (instruct, Claude acts, review output, course-correct) is the core workflow everything else depends on<br>3. Without verification, developers can't tell if their prompts are working -- they need Module 1 skills first to write good prompts, then Module 2 teaches them to close the feedback loop<br>4. Plan mode (Shift+Tab) and /clear are introduced here because they're verification strategies, not prompting strategies		training::developer-path
Explain why TDD comes after the Verification Loop in the developer path, not before or alongside it.	Key points:<br>1. TDD is a specific, strong form of verification -- "the strongest form" per the curriculum. Module 2 teaches the general verification concept; Module 3 specializes it<br>2. A failing test is "the most precise requirement you can give Claude" -- but developers need to understand verification first to appreciate why tests matter<br>3. Tests survive context compaction because they're on disk, making them durable requirements across sessions. This builds on the context awareness from Module 2's /clear discussion<br>4. The Red-Green-Refactor cycle gives Claude concrete pass/fail feedback, which is the verification loop made systematic		training::developer-path
A senior developer pushes back: "Why does the developer path put Context Management at Module 5 instead of earlier? Understanding context seems fundamental." What do you say?	Key points:<br>1. Context management is important but not actionable without the preceding skills. You need to know how to prompt, verify, write tests, and debug before understanding context constraints changes your behavior<br>2. Modules 1-4 give developers practical skills they can use immediately. Context management is optimization -- it makes those skills more efficient<br>3. Module 5 relies on concepts from all prior modules: writing decisions to CLAUDE.md (Module 1 specificity), /compact (Module 2 verification), test files surviving compaction (Module 3), and subagent delegation (preparation for Module 6)<br>4. Teaching context limits too early causes developers to over-optimize before they know what they're optimizing for		training::developer-path
Explain why the Developer Path ends with extensions (subagents, skills, MCP servers) rather than putting them earlier.	Key points:<br>1. Extensions are force multipliers -- they amplify the skills from Modules 1-5 but are useless without them<br>2. Subagents require understanding context management (Module 5) to appreciate the 97.5% context savings<br>3. Skills inject behavior into main context, which only makes sense after understanding what context costs (Module 5)<br>4. The Lens + Reviewer pattern combines delegation with focused depth -- a concept that requires understanding both verification loops and context isolation		training::developer-path
During a training kickoff, a platform engineer asks: "Why does the platform path start with architecture instead of jumping into Terraform and infrastructure setup?" What do you say?	Key points:<br>1. The four-hop request flow (workstation, gateway, VPC endpoint, Bedrock) is the mental model that makes all subsequent decisions coherent<br>2. Without understanding where enforcement points exist (gateway vs. managed settings), you can't make good infrastructure decisions<br>3. The configuration hierarchy (managed > CLI > local > project > user) determines what can be overridden where -- this is prerequisite knowledge for Modules 3-4<br>4. Jumping to Terraform without the architecture mental model produces infrastructure that works but isn't designed for the control points you'll need later		training::platform-path
Explain why the Platform Engineer Path separates Configuration (Module 3) from Permissions (Module 4) instead of combining them.	Key points:<br>1. They solve different problems: configuration routes traffic and sets organizational defaults; permissions control what Claude can do on the developer's machine<br>2. Configuration is about enforcement hierarchy (managed-settings.json overrides everything). Permissions are about rule evaluation (deny > ask > allow)<br>3. Managed settings include non-permission controls like CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC and managed CLAUDE.md. Conflating these with permission rules creates confusion<br>4. The curriculum advice "start permissive, tighten later" applies to permissions but not to configuration -- you want correct configuration from day one		training::platform-path
Explain why Cost Management is Module 5 in the platform path rather than earlier.	Key points:<br>1. You can't manage what you haven't built. Modules 2-4 set up the infrastructure, configuration, and permissions that cost controls depend on<br>2. Model tiering (the primary cost lever) requires the LLM gateway from Module 2 to enforce<br>3. Budget enforcement needs the managed settings from Module 3 (e.g., CLAUDE_CODE_EFFORT_LEVEL)<br>4. Observability (CloudWatch dashboards, allocation tags) makes sense only after you have real traffic flowing through the infrastructure you built in Modules 2-4		training::platform-path
A team lead asks: "Why does the platform path end with Phased Rollout instead of covering it earlier so we can plan?" What do you say?	Key points:<br>1. The rollout plan depends on decisions made in Modules 1-5 -- infrastructure capacity, policy configuration, permission model, cost projections<br>2. Cohort 1 builds the foundation (CLAUDE.md files, org skills, infrastructure validation) -- you need Modules 2-4 knowledge to support them<br>3. Rollback is cohort-level and uses managed-settings.json (Module 3) -- understanding the mechanism before the plan makes the plan concrete<br>4. Success metrics reference infrastructure stability, cost model validation, and developer satisfaction -- all concepts from prior modules		training::platform-path
Explain why the PM training path was restructured from a developer-focused approach to product-first.	Key points:<br>1. The original developer-focused structure taught PMs to understand developers first, which put the cart before the horse<br>2. PMs need Claude Code for their own work (research synthesis, requirements writing, prioritization modeling) before they need technical literacy<br>3. The restructured path has PMs using Claude Code productively within the first module, building confidence and practical skills before tackling technical concepts<br>4. Product-first ordering means a PM who only completes Modules 1-3 still gained direct professional value, whereas the old ordering required completing technical modules before getting to applicable skills		training::pm-path
A PM asks: "Why does the PM path have only 5 modules when developers and platform engineers get 6?" What do you say?	Key points:<br>1. Module count reflects what each role needs, not relative importance. PMs don't need extensions, context management optimization, or infrastructure setup<br>2. The PM path covers distinct territory: research synthesis, prioritization modeling, and requirements writing aren't in the other paths<br>3. Modules 3 and 4 can be taken in either order (both depend on Module 2, Module 5 depends on all four) -- this flexibility wouldn't work with more sequential dependencies<br>4. The path is complete when PMs can write specs that developers (and Claude) can build against and participate in the development process		training::pm-path
Explain why the PM path starts with "How Claude Code Works" before diving into product research.	Key points:<br>1. PMs need a mental model of the tool's capabilities and limitations before they can use it effectively for product work<br>2. Module 1 establishes what PMs can do directly (synthesize data, model priorities, draft specs, prototype) -- setting scope for the entire path<br>3. It surfaces critical limitations upfront: context windows are finite, Claude lacks business context unless provided, and output needs verification<br>4. The "quality of output depends on quality of input" principle from Module 1 underpins everything in Modules 2-5		training::pm-path
Explain why Product Research & Discovery is Module 2 in the PM path and comes before Requirements & Prioritization.	Key points:<br>1. Research produces the raw material that requirements and prioritization operate on -- you can't prioritize what you haven't understood<br>2. The curriculum establishes a discipline: research tells you what the problem is, prototyping tests whether your solution makes sense, and requirements capture what to build<br>3. Module 2 teaches the critical concept of "lens" -- "categorize by workflow stage" vs. "summarize these tickets" produces different output. This applies to all subsequent modules<br>4. The prototyping section explicitly warns against polishing prototypes ("if you find yourself polishing, you've stopped researching and started building") -- this discipline needs to be established before requirements writing		training::pm-path
A PM pushes back: "Why do I need a Technical Literacy module? I'm not writing code." What do you say?	Key points:<br>1. Technical literacy isn't about writing code -- it's about recognizing when technical decisions are being made on your behalf and asking the right questions<br>2. Understanding maintenance cost explains why developers push back on "quick" features. The question "What makes this expensive to change later?" prevents scope conflicts<br>3. YAGNI applies to your roadmap too -- recognizing scope creep from "while we're in there" additions is a product skill<br>4. TDD knowledge means you can write acceptance criteria precise enough to become tests ("POST /login with invalid credentials returns 401" vs. "the user should be able to log in"), directly improving development output		training::pm-path
Explain why Modules 3 (Requirements & Prioritization) and 4 (Technical Literacy) can be taken in either order in the PM path.	Key points:<br>1. They serve different audiences and situations: Module 3 helps PMs do their own work better, Module 4 helps PMs collaborate with developers better<br>2. Both depend on Module 2 (research) but don't depend on each other -- a PM can write requirements without technical depth, and can learn technical literacy without requirements frameworks<br>3. This flexibility accommodates different PM backgrounds: a former engineer might start with Module 3 (less familiar), while a research-focused PM might start with Module 4 (less familiar)<br>4. Module 5 (Working with Development Teams) is where both converge, requiring both skill sets		training::pm-path
During a training kickoff, someone asks: "Why does each path have its own exercise repo instead of sharing one?" What do you say?	Key points:<br>1. Each path's exercises match that role's actual work -- developers practice with a task tracker codebase, platform engineers work with Terraform configs and policy files<br>2. Shared exercises would force roles into irrelevant contexts (PMs don't need Terraform, platform engineers don't need task tracker bugs)<br>3. Exercise repos let learners clone and work independently, with branch-based scenarios (the developer path uses branches like 'feature/task-search' and 'buggy')<br>4. The PM path doesn't have an exercise repo -- its exercises use real product artifacts (support tickets, feature requests, stakeholder discussions) because PM work is context-dependent		training::overview
Explain why the Developer Path teaches debugging after TDD rather than before it.	Key points:<br>1. TDD provides the debugging infrastructure. When you have tests, debugging starts with a failing test -- the most precise symptom possible<br>2. The debugging module explicitly recommends "write a failing test that captures the bug, then share only the test output" -- this technique requires Module 3 skills<br>3. The four-phase framework (understand, reproduce, investigate, fix) relies on the verification loop from Module 2 and the test-writing skills from Module 3<br>4. Teaching debugging before TDD encourages ad-hoc approaches. Teaching it after means developers default to test-first debugging		training::developer-path
A platform engineer asks: "The three-cohort rollout structure seems rigid. Why not just train everyone at once?" What do you say?	Key points:<br>1. Cohort 1 (25 power users) builds the foundation that everyone else depends on -- CLAUDE.md files, org skills, infrastructure validation. These don't exist yet<br>2. Cohort 2 (100 developers) stress-tests what Cohort 1 built at scale -- gateway capacity, cost model accuracy, observability dashboards. Without this, Cohort 3 is flying blind<br>3. Rollback is cohort-level, not all-or-nothing. If a security incident occurs during Cohort 2, you pause 100 users, not 500<br>4. Champions from earlier cohorts provide peer support for later ones -- this scales training without requiring the platform team to hand-hold every developer		training::platform-path
Explain why the PM path's Module 5 (Working with Development Teams) is the capstone rather than being distributed across earlier modules.	Key points:<br>1. Collaboration skills require all four prior modules: understanding the tool (Module 1), research outputs (Module 2), requirements precision (Module 3), and technical literacy (Module 4)<br>2. Module 5 synthesizes: tickets use Module 3 acceptance criteria, session planning uses Module 1 context limits, code review uses Module 4 technical understanding<br>3. Placing collaboration at the end means PMs arrive with something concrete to offer -- not "I want to help" but "here are testable acceptance criteria based on research I synthesized"<br>4. The module explicitly connects PM outputs to developer inputs: "the inputs to that process -- tickets, specs, acceptance criteria -- come from your work"		training::pm-path
What are the four rollback triggers for the phased rollout, and why these specific thresholds?	The four triggers are: (1) security incident of any severity, (2) infrastructure instability (gateway errors above threshold), (3) cost exceeding 25% over projection, (4) more than 40% negative developer feedback. These cover the four failure domains: security (zero tolerance), infrastructure reliability, financial risk, and adoption risk. Each can be measured objectively without subjective judgment.		training::platform-path
Explain the curriculum design difference between how the developer path and platform path teach the same underlying system.	Key points:<br>1. The developer path teaches Claude Code from the inside out: prompting, verification, testing, debugging, context, extensions -- all about the developer-tool interaction<br>2. The platform path teaches from the outside in: request flow, infrastructure, configuration, permissions, cost, rollout -- all about organizational control<br>3. Both paths reference the same system but from opposite perspectives: developers care about context windows for productivity; platform engineers care about context (CLAUDE.md hierarchy) for policy enforcement<br>4. This explains why both paths have 6 modules covering similar territory but with zero overlap in exercises or learning objectives		training::overview
Why does the developer path use a single exercise repo with branches (feature/task-search, buggy) rather than separate repos per module?	Key points:<br>1. A single codebase with branches simulates real development -- developers work on one project across multiple contexts, not isolated toy examples<br>2. Branch-based scenarios let modules build on each other: the task tracker from Module 1's requirements becomes Module 2's buggy implementation becomes Module 3's TDD target<br>3. This matches how Claude Code actually works -- developers navigate branches, context from prior work influences current sessions<br>4. Switching branches with git (not repos) reinforces the /clear discipline from Module 2 -- new branch, new context		training::developer-path
During a training kickoff, a developer asks: "Why does Module 2 specifically teach /clear between tasks?" What do you say?	/clear is taught in Module 2 because it's a verification strategy, not a prompting technique. Accumulated context from a previous task adds noise and wastes tokens. The exercise makes this concrete: after fixing search bugs, you /clear and start due dates. Without /clear, Claude's context is polluted with search-fix details that are irrelevant to the due dates task. It's the context hygiene habit that makes everything in Modules 3-6 more effective.		training::developer-path
Explain why the PM path explicitly warns "prototypes are for learning, not shipping" in Module 2.	Key points:<br>1. PMs using Claude Code for prototyping face a unique temptation: the prototype looks real enough to ship, unlike a sketch or wireframe<br>2. "If you find yourself polishing a prototype, you've stopped researching and started building" -- this discipline boundary is critical because Claude makes it easy to keep iterating<br>3. The warning is placed in Module 2 (research) rather than Module 3 (requirements) because the error happens during discovery, not specification<br>4. Without this discipline, PMs bypass the research-to-requirements pipeline and hand developers prototypes instead of specs, which undermines the entire Module 3-5 progression		training::pm-path
Why does the platform path teach "start permissive, tighten later" for permissions but not for configuration?	Key points:<br>1. Permissions affect developer productivity directly -- overly restrictive rules create friction and workarounds. Starting permissive lets you observe real usage patterns before restricting<br>2. Configuration (traffic routing, managed CLAUDE.md, non-essential traffic) should be correct from day one because it controls where data flows and what baseline behavior exists<br>3. allowManagedPermissionRulesOnly starts false so teams can define their own rules; switching to true later is straightforward for regulated environments<br>4. This distinction teaches platform engineers to separate "what must be enforced immediately" (data flow) from "what should be tuned based on evidence" (permission granularity)		training::platform-path
Explain why the PM path teaches "the lens determines what Claude finds" as a key concept in Module 2.	Key points:<br>1. "Categorize by workflow stage" produces different output than "summarize these tickets" -- the framing question controls what Claude surfaces<br>2. This is the PM-specific version of prompt specificity (Developer Path Module 1), tailored to research rather than code generation<br>3. The concept transfers to every subsequent module: framing prioritization criteria (Module 3), framing architecture questions (Module 4), framing acceptance criteria (Module 5)<br>4. It teaches PMs that Claude is a lens, not an oracle -- it finds what you point it at, so the skill is in choosing where to point		training::pm-path
How do all three training paths handle the relationship between their paths? Is there cross-referencing?	Each path ends with a "What's Next" section that points to the other two paths for complementary skills. Developer path points to platform path for team adoption. Platform path points to developer path for individual skills and PM path for product collaboration. PM path points to both others plus the Product Development section. Cross-referencing acknowledges that roles interact without forcing anyone through content designed for a different role.		training::overview
Explain why the developer path's Module 3 specifically warns against tests that only exercise mock behavior.	Key points:<br>1. Mocking is the most common way TDD goes wrong with Claude Code -- Claude will happily generate mock-heavy tests that pass but test nothing real<br>2. The exercise makes this concrete: rewrite TaskServiceMockTest.java from Mockito mocks to @SpringBootTest with real dependencies, then compare what each catches<br>3. The curriculum states "if you mock a dependency, make sure the test still validates real logic in your code" -- this is a judgment call that requires the verification mindset from Module 2<br>4. "Never delete a failing test" is paired with the anti-mock guidance because developers often delete mock tests that break during refactoring, hiding real issues		training::developer-path
