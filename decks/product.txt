What is product thinking?	The practice of asking -- before and while you write code -- whether the thing you're building connects to something a user actually needs. It's questioning what you're building and why, before optimizing how.	product::product-thinking

They implemented synonym matching and fuzzy search plus tracking, while Engineer A built a technically correct but practically irrelevant trie-based autocomplete. The difference: a 30-minute investigation before the first line of code.	product::product-thinking

Explain the three requirement formats and when to use each in combination.	User stories ("As a [role], I want [action] so that [outcome]") communicate intent to stakeholders. Job stories ("When [situation], I want [motivation] so I can [outcome]") surface triggering context and urgency. Acceptance criteria ("Given [context], when [action], then [result]") define verifiable outcomes. Use them together: job stories capture context (3am urgency), user stories scope features, acceptance criteria verify implementation.	product::requirements
What makes a requirement good enough to serve as a Claude Code build prompt?	It must specify: concrete inputs and outputs, constraints, and verification criteria. "Add rate limiting to the login endpoint" becomes a good prompt when it specifies: which endpoint (POST /api/auth/login), what limit (5 attempts per email per 15 minutes), what response (429 with Retry-After header), and boundary behavior (6th attempt blocked, counter resets after 15 minutes). Humans fill gaps with institutional knowledge; Claude Code fills gaps with assumptions.	product::requirements
What is a vertical slice and why does it matter for Claude Code-assisted development?	A vertical slice delivers complete user-facing behavior touching every layer needed (UI, API, logic, data). "User can roll back a deployment" is a vertical slice. "Build all the database models" is a horizontal layer. Vertical slices reduce integration risk because each slice works end-to-end. For Claude Code, vertical slices make better build prompts because each one is independently verifiable.	product::requirements

What is a prototype in the wiki's definition and what is its primary output?	A prototype is a question in code form -- a time-boxed experiment that confirms or invalidates a specific hypothesis. In the deployment rollback prototype example, what did the team learn and how did it change their plan?	Health check detection time: min 18s, median 24s, max 38s. The 38-second worst case left only 22 seconds for rollback, but existing deploys took 45-90 seconds. The hypothesis was partially invalidated: detection was fast enough, but combined detection + rollback exceeded the 60-second target. What is "prototype rot" and why is it the most common prototyping failure?	"Let's just clean this up and ship it." The code was written without error handling, tests, or architectural consideration. Cleaning it up takes as long as rewriting it, and the result carries structural decisions made under a throwaway mindset. If the prototype is genuinely close to production quality, you were overbuilding the prototype -- next time, build less.	product::prototyping
Explain why you should never prototype without a hypothesis.	Without a hypothesis, you're doing open-ended exploration with no success criteria. What are the four factors in the RICE scoring framework and the formula?	Reach: how many users affected (concrete number, not percentage).<br>Impact: how much per user (3=massive, 2=high, 1=medium, 0.5=low).<br>Confidence: how sure are you (100%=data-backed, 80%=educated guess, 50%=speculation).<br>Effort: person-weeks including design, testing, docs.<br>Formula: RICE = (Reach x Impact x Confidence) / Effort	product::prioritization
When does RICE scoring break down?	Three situations:<br>1. Confidence is consistently low across all features -- RICE degenerates into reach-times-impact; invest in research before scoring<br>2. Strategic features with future reach -- RICE doesn't model unlocking a market segment; argue these on strategy, not score<br>3. Effort estimates are wildly uncertain (2 to 12 weeks) -- do a spike to narrow the estimate before scoring	product::prioritization
What is opportunity cost and how do you calculate it for a feature decision?	Opportunity cost is what you give up by choosing one option over another. Calculate it: list top 3 alternatives, estimate the value each delivers, and the opportunity cost of your proposed feature is the highest value among those alternatives. Example: a dashboard estimated at $40K ARR has an opportunity cost of $120K ARR if the alternative is retaining 15 at-risk enterprise accounts.	product::prioritization
How does Claude Code help model opportunity cost between competing features?	Describe both features with effort, reach, revenue estimates, and risks. Ask Claude Code to model the trade-off: what's the opportunity cost of A over B? What changes if effort estimates shift by 50%? Include sensitivity analysis on conversion assumptions. Claude Code shows which assumptions carry the most weight and where the decision flips -- it won't decide for you, but it makes the bets explicit.	product::prioritization
What is cost of delay and when should it override RICE scores?	Cost of delay measures what you lose per unit of time by not building a feature. A feature worth $25K that becomes worthless in 6 weeks is more urgent than one worth $50K that holds its value for 6 months. Features with high cost of delay should jump the priority queue even if their RICE score is lower. Examples: regulatory deadlines (binary), competitive responses (weekly decay), seasonal campaigns (total loss after date).	product::prioritization
What are the three forms of "no" in prioritization?	"Not now" -- feature has merit but ranks lower; capture rationale for re-evaluation next cycle.<br>"Not this way" -- problem is real but solution is wrong; go back to research (e.g., weekly email digest instead of 6-week custom dashboard).<br>"Not ever" -- contradicts product direction, targets wrong segment, or has permanently unfavorable economics. Be direct.	product::prioritization
How should a "no" decision be documented to prevent repeated re-pitching?	Record: date, status (deferred to when), context (who requested, effort estimate), why not now (RICE score, opportunity cost, workarounds available), and revisit conditions (specific triggers like "request volume exceeds 25 accounts" or "workaround becomes infeasible"). This document is the answer next time someone asks -- it prevents the same debate every sprint.	product::prioritization
